# UX Writing

## Button Labels

Use specific verb + object. "Save changes" not "OK." "Delete message" not "Yes."

| Avoid | Use instead |
|-------|-------------|
| OK, Yes, No | Save changes, Delete message, Discard draft |
| Submit | Create account, Send message, Publish post |
| Click here | View pricing plans, Read the docs |

## Error Messages

Answer all three: what happened, why, how to fix.

❌ `"Invalid input."`
✅ `"Email address isn't valid. Please include an @ symbol."`

## Don't Blame Users

❌ `"You entered an invalid date."`
✅ `"Please enter a date in MM/DD/YYYY format."`

## Voice vs Tone

**Voice** stays consistent (always professional, always friendly). **Tone** adapts to context (empathetic for errors — never humorous).

## Empty States

Treat as onboarding moments:
1. Acknowledge: "No projects yet"
2. Explain value: "Projects help you organize related work"
3. Provide action: "Create your first project"

## Terminology

Pick one term and stick with it throughout. "Delete" everywhere — not "Remove," "Trash," or "Remove permanently."

## Accessibility

Links must stand alone:
❌ `"Click here for more information"`
✅ `"Read more about our pricing plans"`

## Translation

- Plan for text expansion (German +30%, Japanese -30%)
- Keep numbers separate from sentences
- Avoid idioms and culture-specific references
- Use consistent punctuation

## Confirmation Dialogs

Name the action and consequences:
- "Delete project" / "Keep project" (not "Yes" / "No")
- "Remove member" / "Cancel" (not "OK" / "Cancel")

The user should know exactly what they're confirming without reading surrounding context.
